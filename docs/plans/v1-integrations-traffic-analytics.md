# v1 — First-party data: GSC/GA4/Bing integrations + Traffic + LLM Analytics

Detailed implementation plan. Specs: `docs/roadmap/integrations.md`, `docs/roadmap/traffic.md`, `docs/roadmap/llm-analytics.md`. Rules: `Agents.md` + `docs/invariants.md` (all 12 apply). Repo: `/code/abhij1306/Searchify`.

## 0. Setup, scope, decisions

-   **Branch:** `git fetch origin` then branch `vorflux/integrations-traffic-analytics` from the **latest** `origin/main` (user directive — never a stale local main).
-   **In scope:** integrations backend (GSC + GA4 on one shared Google grant; Bing on a Microsoft grant), Traffic surface (backend + `/traffic` screen), LLM Analytics surface (backend + `/analytics` screen), Settings → Integrations UI.
-   **Out of scope (non-goals):** S3 payload offload (inline JSONB with a config size cap this pass), arbitrary custom `from`/`to` windows (read endpoints serve persisted snapshot windows only), server-log/crawler ingest (Release 1.3), live OAuth E2E (needs real client credentials — wired into env later), all other roadmap surfaces.
-   **Testing:** fully mocked — fake OAuth server + recorded provider fixtures via `httpx.MockTransport` (established pattern: `connectors/discovery_models/factory.py`, `tests/component/test_site_health_worker.py`). Assert no token in any DTO/log.
-   **Backend setup caveat:** run `uv sync` once before any backend verify command — `uv run pytest` / `uv run ruff check .` need the `dev` optional-dependencies installed. All backend commands run from `backend/`.
-   **Migrations:** greenfield — edit models, register in `app/models/__init__.py`, `uv run alembic downgrade base && uv run alembic upgrade head`. **No new revision files.** Migrations live at repo-root `migrations/` (single revision `migrations/versions/0001_initial.py`; `backend/alembic.ini` script_location = `../migrations`); run alembic from `backend/`.
-   Lint gate for every backend task: `uv run ruff check .` (from `backend/`, after `uv sync`). Frontend verify: `pnpm test -- <file>`, `pnpm lint`, `pnpm build` (from `frontend/`; **pnpm only**).

## 1. Grounding reconciliations (coded reality beats spec wording)

1.  `TaskQueue[T]` + `PostgresQueueSpec` **already exist** (serving AuditTask / SiteCrawlTask / ContentGeneration). Remaining work is type-only TypeVar widening (PEP 695 constraint tuples in `app/core/config/task_queue.py` **and** `app/orchestration/postgres_task_queue.py` + its TYPE_CHECKING imports; the `TaskQueue[T]` Protocol in `orchestration/task_queue.py` is unconstrained — no change) + two new spec constants; guarded by existing queue regression tests.
2.  "PageArtifact" was never coded → page join targets `SiteUrl` (`uq_site_url_project_hash`) via `domain/site_health/normalization.canonical_identity` (the one canonical-form owner — inv. 2), not `analysis/normalization.py`.
3.  `ReferralEvent.import_id` → `IntegrationImportArtifact` (CASCADE). No `TrafficImport` table is created; events also carry `source_metric_row_id` + `sanitize_version` (inv. 4). Unique `(import_id, content_hash)` dedupes.
4.  Nav was simplified in PR #14: no disabled "soon" machinery — **add** entries to `NAV_GROUPS` (Analyze group), icons to the canonical `lib/icons.ts` map, titles to `top-bar.tsx` `PAGE_TITLES`. No `live` flag exists.
5.  Settings → Integrations is a **4th tab** in `settings-screen.tsx` (`?tab=integrations` deep-link = OAuth callback landing), not a nav item.
6.  Integrations OAuth start/callback are 302 endpoints → full-page navigation via `lib/navigate.ts` `assignLocation` (never `apiClient` fetch, never the auth `oauth-buttons.tsx` fetch-start pattern).
7.  No recurring scheduler exists — the integrations dispatcher (I10) is the first; future recurring audits reuse it (not the reverse).
8.  `app/api/oauth.py` + `security.py create_oauth_state/decode_oauth_state` are the auth **sign-in** scaffold; integrations OAuth is a new router that **extends** those helpers (optional workspace/user/jti claims; session_nonce cookie comparison skipped for integration states — they bind via persisted claims + `get_current_user`) — no duplicate state helper (inv. 2).

## 2. Cross-workstream contracts (pin before dependent tasks start)

-   **C1 — GA4 dimension packing:** config-owned dataset → dimensions/metrics templates (I1) declare GA4 datasets carrying channel / source-medium / referrer / landing dims. **Pinned here (authoritative for both workstreams, so A1 needs no I1 output):** GA4 dataset ids — `ga4_channel_daily` (dims: sessionDefaultChannelGroup, date), `ga4_source_medium_daily` (dims: sessionSource, sessionMedium, date), `ga4_referrer_daily` (dims: fullReferrer, date), `ga4_landing_daily` (dims: landingPage, sessionSource, sessionMedium, date); all carry sessions/engagedSessions/conversions metrics. `dimension_key` packs multi-dim rows by joining the dimension values **in declared template order with `" | "`** (single-dim rows use the bare value). Analytics/traffic consume: referral ingest (A5) reads `ga4_referrer_daily` + `ga4_source_medium_daily` (= `TRAFFIC_GA4_REFERRAL_DATASETS`); traffic inclusion (A7) reads `ga4_channel_daily` + classifier match on `ga4_source_medium_daily`/`ga4_landing_daily` dims; page stats read `ga4_landing_daily`. (integrations owns the templates; analytics consumes.)
-   **C2 — OAuth callback 302 target:** `/settings?tab=integrations` with `connected=<provider>` / `error=<code>` query params the panel reads for its result notice.
-   **C3 —** `POST /projects/{id}/traffic/sync` **response:** `{sync_run_id, connection_id, status}` per queued run (frontend polls `GET /integrations/{id}/syncs/{sync_id}`).
-   **C4 — paged envelopes:** keyset `next_cursor` (site-health convention) for traffic pages/queries + analytics referrals.
-   **C5 — post-sync hook + referral chain:** the integrations worker calls `domain/analytics/enqueue.enqueue_post_sync_projections()` after derivation, which enqueues, per import artifact, the chained lifecycle `ingest_referrals` → (on completion) `classify_referrals` → (on completion) `analytics_snapshot_refresh`, plus `traffic_snapshot_refresh` for the affected windows. `traffic/sync` calls the integrations enqueue service. Both directions monkeypatched/fake-tested, and the full chain has one lifecycle test (A6).
-   **C6 — schema reconcile:** backend DTOs are source of truth; frontend diffs `schemas.ts` against each landed API before marking a screen done.

Intra-wave `after` markers gate task pickup within a wave (e.g. A4 starts only once A1 lands; F7 after F6, F9 after F8, I9 after I8) — waves are dispatch batches, not promises that every member starts simultaneously. **Shared edit targets within a wave are serialized/rebased, never edited in true parallel:** `backend/app/models/__init__.py` (I2, A2, A5), `Settings` in `config/__init__.py` (I1, A1), `app/main.py` router registration (A9, A10, I3's integrations router), `infra/docker/docker-compose.yml` (I6, I10, A3), frontend `schemas.ts` + the query-keys facade (F1–F3), `settings-screen.tsx` (F5). Where two in-wave tasks touch one file, the second dispatched rebases on the first's commit.

Land as up to three sequential PRs — (1) integrations backend, (2) traffic+analytics backend, (3) frontend — each branched/rebased onto fresh `origin/main` as the previous merges (per git-pr-workflow skill).
